  **A. Fix check.ts rounding (fixes Issue 5 — check count):**
  The current allocation uses `Math.round` per weight then adjusts drift on `allocations[0]`. Problem: when a sub-generator (like build) fails to produce items because entities lack required capabilities, the total falls short of `step.count`.

  Fix the rounding to use a largest-remainder method that guarantees exact sum:
  1. Compute `Math.floor(totalCount * weight)` for each allocation
  2. Calculate remainder = `totalCount - sum(floors)`
  3. Sort allocations by fractional part descending (the ones closest to rounding up)
  4. Add 1 to the top `remainder` allocations
  This guarantees allocations sum exactly to `totalCount`.

  Additionally, after generating sub-items, if the total is less than `step.count` (because a sub-generator returned fewer items than allocated), redistribute the shortfall to other generators that CAN produce more. Specifically:
  1. After the generation loop, check if `allItems.length < totalCount`
  2. If short, calculate deficit = `totalCount - allItems.length`
  3. Try to generate more `read` items first (most reliable generator), then `choose`, then `hear`
  4. This ensures check steps always hit their target count when entities support it
